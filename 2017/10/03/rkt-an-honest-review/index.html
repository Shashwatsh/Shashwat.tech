rkt: an honest review

Shashwat's Blog
2017-10-03
containers, virtualization, rkt, docker alternative

tl;dr

rkt is production ready but it’s not mature enough to become a replacement for docker or other container runtimes as of october 2017.

last december i got sick of docker and started looking for a better alternative.

i found rkt, at first i was scared to use it since it’s just 6 months old but after reading alot of good reviews and thinking, i came to an conclusion to give it a try, i had absolutely no problems with it. although, installing it on debian jessie is not preferable by me but that is just a little container issues and has nothing to do with rkt’s stability.

rkt is stable and ready for production but comes with it’s own problems (depending on what you’re trying to do) but first i’m gonna discuss the pros and what i liked about it.
building container images are much easier than before since i don’t need a base image and dockerfiles. rkt also has a simpler and more secure architecture than docker and you can run your existing docker image with rkt just by adding docker:// prefix to image repo url (hint: docker://alpine-sh).

see AppC spec: https://github.com/appc/spec

one downside with rkt is that it lacks alot of features unlike docker, for me not having a –d option was a fairly important feature (though, you can easily do it with systemd), lack of an API for managing pods is another (it has grpc api which can do nothing more than listing pods). third is that pods cannot be restarted (this is by design, see https://coreos.com/rkt/docs/latest/devel/pod-lifecycle.html) alternatively, you can export pod’s rootfs or create another pod with the same image.

hopefully, you won’t get any problems if you’re using k8s with rkt (rktnetes) but be careful. support for rktnetes (k8s with rkt) is limited.

conclusion: rkt is good and stable but it can’t compete with other container runtime right now.

links\: rkt vs other container runtimes: https://coreos.com/rkt/docs/latest/rkt-vs-other-projects.html
